# Explain Code Skill

코드를 설명할 때 항상 다음을 포함하세요:

## 1. 유추로 시작
코드를 일상 생활의 무언가와 비교합니다. 사용자가 이미 알고 있는 개념을 활용하여 새로운 개념을 설명합니다.

**예시:**
- "이 함수는 레스토랑의 주문 시스템과 같습니다..."
- "이 클래스는 자동차 공장의 청사진과 비교할 수 있습니다..."
- "이 데이터 흐름은 우편 배달 시스템처럼 작동합니다..."

## 2. 다이어그램 그리기
ASCII 아트를 사용하여 흐름, 구조, 또는 관계를 시각적으로 표시합니다.

**다이어그램 유형:**
- **데이터 흐름:**
  ```
  Input → Process → Output
  ```
- **계층 구조:**
  ```
  Parent
  ├── Child 1
  └── Child 2
  ```
- **순차적 프로세스:**
  ```
  Step 1 → Step 2 → Step 3
     ↓        ↓        ↓
  Result1  Result2  Result3
  ```
- **순환/루프:**
  ```
  Start → Check → Process
            ↑         ↓
            └─────────┘
  ```

## 3. 코드 단계별 설명
무슨 일이 일어나는지 단계별로 명확하게 설명합니다.

**구조:**
1. **먼저 무엇이 발생하는가?**
2. **그 다음은?**
3. **마지막 단계는?**
4. **결과는 무엇인가?**

각 단계에서:
- 변수가 어떻게 변하는지
- 조건이 어떻게 평가되는지
- 함수 호출이 무엇을 반환하는지

## 4. 주의할 점
일반적인 실수나 오해를 명시합니다.

**포함할 내용:**
- ⚠️ 흔한 버그나 실수
- 💡 성능 고려사항
- 🔍 디버깅 팁
- ❌ 피해야 할 안티패턴
- ✅ 베스트 프랙티스

## 스타일 가이드

### ✓ 좋은 예시:
- 대화체 사용: "이 코드는 먼저 ~를 확인합니다"
- 구체적인 예시 제공
- 복잡한 개념에는 여러 유추 사용
- 코드 조각을 인용하며 설명

### ✗ 피해야 할 것:
- 지나치게 기술적인 용어만 사용
- 다이어그램 없이 긴 텍스트로만 설명
- 추상적인 설명만 제공
- 사용자의 수준을 고려하지 않음

## 템플릿

```markdown
## 🎯 유추

[일상생활 비교]

## 📊 구조 다이어그램

[ASCII 다이어그램]

## 🔄 동작 방식

1. **첫 번째 단계**: [설명]
2. **두 번째 단계**: [설명]
3. **세 번째 단계**: [설명]

## ⚠️ 주의할 점

- [일반적인 실수 1]
- [일반적인 실수 2]
- [베스트 프랙티스]

## 💡 실전 예시

[구체적인 사용 예시]
```

## 복잡도 수준별 접근

### 초급 (Beginner):
- 가장 단순한 유추 사용
- 기본 개념부터 설명
- 더 많은 다이어그램과 예시

### 중급 (Intermediate):
- 기술적 세부사항 추가
- 디자인 패턴 연결
- 성능 고려사항 포함

### 고급 (Advanced):
- 아키텍처 관점 설명
- 트레이드오프 분석
- 최적화 전략
